personal blog | Taking a Close Look at Game Design with Game Programming Patterns🎮
A couple of days back, I started to read Game Programming Patterns by Robert Nystrom, and it has changed the game for me! This book has shown me essential insights on how to design strong and expandable game systems. Here's what I've picked up so far:

Before I begin, if you want to read the book too here is the free online version of it - https://lnkd.in/d2cUH9fp

📜 Main Patterns I Looked Into:

1. Command Pattern: 

It separates game input from actions putting requests into objects. This has helped me a lot to make my game controls easy to change and set up. For instance, player inputs like "jump" or "shoot" can now be switched or even saved for later use without having to rewrite the logic. It's great for creating flexible and reusable input systems.

2. Observer Pattern:
This approach makes sure that things like UI elements respond on their own when the game's status changes. I don't have to hard-wire communication between systems anymore. If a player's health goes down, the UI refreshes without linking the two systems. It adds a fresh degree of separation and growth potential to my work.

3. State Pattern: 
Game characters often switch behaviors, like idle, attack, or run states. The State pattern keeps this tidy and easy to manage. By making each state its own class, I've gotten rid of the messy sprawling if-else logic. This keeps my code neat and easy to expand.

4. Component Pattern:
We can build game objects, like characters or items using separate components (e.g., health movement). Instead of making huge, all-in-one classes, I now put objects together by mixing various components. This makes the code cleaner and easier to keep up. This pattern has helped me create complex behaviors from simple reusable parts.

5. Game Loop: 
This forms the core of any game. This pattern makes sure that updates to game objects handling input, and drawing on screen happen in order every frame. I've learned how crucial it is to keep things running and to control how the game flows.

6. Flyweight Pattern:
Optimizing memory plays a crucial role in games that have lots of objects. The Flyweight pattern gives me the ability to share data among similar objects such as bullets or enemies, without copying memory. It's a clever technique to conserve resources while maintaining high performance.

Why You Should Read This Book:

If you’re into game development, this book is a must-read. It’s practical, clear, and applies directly to the challenges we face when building games. Whether you’re handling AI, UI, or game mechanics, the patterns in this book will simplify your development process and make your code more flexible and maintainable.



The journey into design patterns isn’t just about writing code; it’s about writing smarter, future-proof code. 🛠️

Your allies are light, sound, and stealth; your mistakes cause the house to change. Deal with growing dangers and choose between making amends or going deeper into the shadows.

Heavily Inspired by LSplash's "Doors"

                                                        

Features
Each run creates a new room, but some offer set high-stakes moments. Procedural horror with handcrafted beats.

Anomalies that take into account player behavior, room state, and progression to generate significant tension are known as context-aware anomalies.

Multiple endings can result from moral decisions and found memory fragments that impact abnormal behavior in replayable narrative paths.


                                                                                  

Anomalies
There are things in the house that seem like fears given a shape. Anticipate the following:

- distort your perceptions or temporarily make movement feel incorrect;

Reward keeping a room lit or punish lingering in the dark;

- pose as doors or useful objects, but their true purpose is revealed;

- stalk with only sound as a clue from the outskirts of sight;

React when the surroundings are overstimulated (too much light, too many noises).

Most traffic games ask you to stop cars or build roads. We asked a different question: What if you could just delete the traffic? 😈

We are currently prototyping the core mechanics for Terrific Traffic, a hypercasual arcade game where you play as a "Chaos Manager."

The Problem: Blind drivers at a 4-way intersection.
The Solution: You can't stop them. You can only TAP TO DELETE them.
